# SOLA v5.3.28

CI fix for v5.3.27. Two of the three new Behat scenarios that shipped in v5.3.27 failed in CI; this release fixes them.

## What broke

The two new `@javascript` drawer scenarios in `tests/behat/chat_widget.feature`:

- **Drawer renders conversation starter buttons on open** — `.local-ai-course-assistant__starters` "is not visible".
- **Close button closes the drawer** — `Element click intercepted... Other element would receive the click: <div class="aica-consent-banner">`.

Both failures had the same root cause. The first-run SOLA consent banner (added to the chat widget mustache for GDPR/FERPA compliance) is rendered with `position:absolute; inset:0; z-index:20; background:rgba(255,255,255,0.98)` — it covers the entire drawer interior until the learner clicks **I accept**. The existing escape-key scenario passed because it dispatches a keyboard event, not a click; the two new scenarios click DOM nodes inside the drawer and the consent banner intercepted those clicks.

The third new scenario (the GDPR delete-all confirm flow on `gdpr_self_service.feature`) passed in CI — it doesn't open the drawer.

## What's new

`tests/behat/chat_widget.feature` Background pre-grants the `aica_sola_consent_given` user preference for `student1` and `teacher1`:

```gherkin
And the following "user preferences" exist:
  | user     | preference              | value |
  | student1 | aica_sola_consent_given | 1     |
  | teacher1 | aica_sola_consent_given | 1     |
```

The preference is checked at `classes/hook_callbacks.php:754` and gates the `{{^consentgiven}}` block in `templates/chat_widget.mustache:305`. Pre-granting it skips the banner rendering entirely in the test environment, which is the right behaviour for Behat scenarios that need to interact with drawer interior elements.

This is a test-fixture change, not a feature change. Production users still see the consent banner on first drawer open until they explicitly accept.
