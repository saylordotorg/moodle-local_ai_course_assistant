# SOLA v6.0.0 — Release Notes, Feature Summary, and Admin Walkthrough

## Part 1: Release Notes (10 June 2026)

**Operational-maturity milestone.** Major version bump reflecting that the v5.11 → v5.12 → v5.13 sequence (vendor-rec optimizations, premium escalation routing, defensive defaults, kill-switch fix) is now production-grade. v6.0.0 ships the last piece of the "leave it alone" safety net: a built-in daily cost-anomaly detector that catches spend spikes the existing 80% / 95% / 100% threshold alerts miss. Plus a self-test CLI for verifying the alert delivery path before relying on it. No schema change; no breaking changes from v5.13.

- **New: cost anomaly detector (`classes/cost_anomaly_detector.php` + `classes/task/cost_anomaly_check.php`).** Daily scheduled task that aggregates today's site-wide SOLA spend in USD (tokens × rate-card via `token_cost_manager`), computes the rolling 7-day median, and emails the `spend_notify_emails` recipient list when today exceeds the configured multiplier × median. Default multiplier is 2.0; tunable to 1.5 (more sensitive) or 3.0 (less sensitive). Off by default; opt in via `cost_anomaly_enabled`. Idempotent per day (won't re-send if cron runs twice). First 7 days after enabling produce `insufficient_history` status (no historical baseline) and never alert. Catches three failure modes the existing spend-cap thresholds miss: (1) runaway course where absolute spend stays under the ceiling but a single course suddenly drives 10x its usual volume, (2) accidental premium-tier enable, (3) provider misroute (e.g. failover chain sending chat to Claude Haiku at $0.39/learner/mo instead of Gemini Flash at $0.042). The alert email includes the top 10 courses by spend that day so admins can identify the source immediately. Scheduled at 09:05 every day; admins can adjust via the standard Moodle scheduled-tasks UI. This is the in-SOLA equivalent of the Redash query at `.drafts/sola-redash-cost-anomaly-2026-06-09.md`; sites without Redash get the same coverage built-in.
- **New: spend-alert delivery self-test CLI (`admin/cli/send_spend_alert_test_email.php`).** Sends a test email through the same recipient resolution and delivery path used by `spend_guard::check()` (80/95/100% thresholds, existing) and `cost_anomaly_detector::maybe_send_alert()` (v6.0). Lets admins verify the alert path BEFORE relying on it instead of waiting for a real spend spike to discover the recipient list is misconfigured. Supports `--dry-run` to print the resolved recipients and email body without sending. Per-recipient delivery result on stdout; non-zero exit on any send failure.
- **Bug fix carried over from v5.13.0:** `emergency_control --chat` is now a real kill switch (it was a silent no-op from v5.4.5 through v5.12.x because it wrote `spend_cap_site=0` thinking 0=paused but `spend_guard::get_cap()` treats 0 as unlimited). The dedicated `emergency_chat_disabled` flag added in v5.13 carries forward unchanged.
- **Carried over from v5.13.0:** `spend_cap_per_course_default` (default per-course cap fallback) and the Saylor-defaults Mistral removal.
- **Carried over from v5.11–v5.12:** mastery classifier routing off chat tier, Voyage embeddings + opt-in rerank, OpenAI cache-hit visibility, Claude Opus 4.7+ temperature deny-list, premium escalation router (Opus 4.8 for ~5% of multi-step STEM turns).
- **`spend_notify_emails` set on dev** to `tom.caswell@saylor.org` (replaces the fall-through to all 8 site admins). Configured via SSM; not part of the v6.0.0 code itself.
- **14 new PHPUnit tests** (`tests/cost_anomaly_detector_test.php`) covering the pure functions (median, day-start arithmetic), the state machine (disabled / insufficient-history / ok / anomaly branches), admin-multiplier handling (respects setting, clamps below-1.0 to default), daily-spend aggregation with a synthetic msgs row, and the per-day idempotency flag for `maybe_send_alert`.

### Why v6.0.0 (not v5.14.0)?

Major version bump because the v5.11 → v6.0 sequence taken together is the "production-grade operational maturity" milestone for SOLA. From v5.10.0 backward, SOLA was feature-complete for the universal-rollout pilot but lacked the cost-discipline safety net that 100K-MAU production needs. With v6.0:

- **Cost** picks are settled per the 2026-06-09 vendor doc (~$1,370/mo at 100K Saylor MAU baseline + ~$700/mo premium tier at 5% escalation, optional).
- **Safety** has every layer: per-course default cap (v5.13) + working kill switch (v5.13 fix) + daily cost-anomaly detector (v6.0) + 80/95/100% threshold alerts (existing) + spend-alert self-test (v6.0).
- **Quality** has the A.10 bake-off resolved (Opus 4.8 as premium escalation), Voyage embeddings + rerank shipped (opt-in), classifier off chat tier.

The version-number jump signals to downstream consumers that SOLA is past the iterative-feature phase and into the operational-maturity phase. No breaking changes — a v5.x site upgrades to v6.0 transparently.

### Backwards compatibility

`cost_anomaly_enabled` defaults to `0` (off). Existing sites upgrade to v6.0.0 with no behaviour change. Opt in to the daily detector by checking the box in **Site administration → Plugins → Local plugins → AI Course Assistant → Cost anomaly detector (v6.0)**.

## Part 3: Admin Walkthrough (~12 min)

Audience: admin standing up SOLA's operational-maturity defensive defaults end-to-end on a fresh v6.0.0 install (or upgrading from v5.x).

### 0. Setup (1 min)

Have admin access to Site administration → Plugins → Local plugins → AI Course Assistant. After upgrading from v5.x, run `php admin/cli/upgrade.php` to apply v6.0 setting defaults. Confirm the upgrade exit code is 0.

### 1. Configure the spend-alert recipient list (2 min)

Site administration → Plugins → Local plugins → AI Course Assistant → **Spend caps** group → **Spend notification emails** (`spend_notify_emails`). Set a comma-separated list of recipients who should receive every cost alert (cap thresholds + the v6.0 daily anomaly check). Blank falls back to ALL site admins, which is rarely what you want at scale.

Saylor recommendation for dev: `tom.caswell@saylor.org` (already set as of 2026-06-10). For Saylor production: a 2-3-person on-call list.

### 2. Verify the alert delivery path BEFORE relying on it (2 min)

From the admin shell:

```bash
php local/ai_course_assistant/admin/cli/send_spend_alert_test_email.php --dry-run
```

Confirms the resolved recipient list and the email body without sending. Then for real:

```bash
php local/ai_course_assistant/admin/cli/send_spend_alert_test_email.php
```

Check inbox(es) for "[SOLA spend guard] TEST email — verify alert delivery path". If the email lands, the alert path is live and the v6.0 cost-anomaly detector + the existing v5.x cap thresholds will both deliver when they fire. If it doesn't, troubleshoot Moodle's SMTP config (Site administration → Server → Email → Outgoing mail configuration) before turning on the detector.

### 3. Set the per-course default cap (1 min)

Site administration → Plugins → Local plugins → AI Course Assistant → **Spend caps** group → **Default per-course spend cap (USD)** (`spend_cap_per_course_default`). Set to `30` (or whatever defensive ceiling you want per course per month). Save.

This is the v5.13 fallback: every course without an explicit per-course override gets this cap. The existing 80% / 95% / 100% notification email pipeline (to the recipients you set in step 1) carries through unchanged.

Saylor dev value as of 2026-06-09: `$30/mo per course`. Course-level overrides (via the course-settings UI) beat the default.

### 4. Enable the daily cost anomaly detector (2 min)

Site administration → Plugins → Local plugins → AI Course Assistant → **Cost anomaly detector (v6.0)** group. Check **Enable cost anomaly detector** (`cost_anomaly_enabled`). Leave **Anomaly multiplier** at the default `2.0` (today's spend must exceed 2.0 × the 7-day median to alert). Save.

The detector runs nightly at 09:05 via Moodle cron. The first 7 days after enabling produce `insufficient_history` status and do not alert — that's expected; the detector needs a baseline. After 7 days, you'll start seeing real evaluations in the cron log:

```
cost_anomaly_check: status=ok today=$1.2345 median=$1.1111 ratio=1.11x threshold=2.00x
```

When the ratio exceeds the threshold, an email goes to your `spend_notify_emails` recipients with the top 10 courses by spend that day. Idempotent per day: a flag in `config_plugins` stops duplicate emails if cron runs the task more than once.

If you want earlier warnings at the cost of more false positives during enrollment bursts, lower the multiplier to `1.5`. If your usage is bursty enough that 2x spikes are routine, raise to `3.0`.

### 5. Verify the kill switch (2 min)

Confirm `emergency_control --chat` actually blocks chat traffic (the v5.13 fix). From the admin shell:

```bash
php local/ai_course_assistant/admin/cli/emergency_disable.php --chat --reason="v6.0 verification smoke"
```

CLI should print `emergency_chat_disabled (set to 1)`. Open any SOLA-enabled course's chat drawer — it should show the "SOLA is paused" message instead of a real chat response. Then restore:

```bash
php local/ai_course_assistant/admin/cli/emergency_disable.php --restore --chat
```

Chat resumes. The flag is unset on restore.

If you skipped this step on prior releases (v5.4.5 through v5.12.x), you wouldn't have noticed the silent no-op because the round-trip didn't visibly change behaviour. v6.0 inherits the v5.13 fix; this is now a real kill switch.

### 6. Confirm the failover chain (1 min)

Site administration → Plugins → Local plugins → AI Course Assistant → **Spend caps** → **Failover chain**. Saylor default as of 2026-06-09: `chat:openai`. With `failover_per_call_enabled = 1`, any Vertex AI Gemini failure on a chat call falls through to OpenAI gpt-4o-mini transparently.

Mistral was originally specified as the EU-resident ultimate fallback but was removed from Saylor's chain on 2026-06-09 pending training-opt-out + ZDR. Provider class stays available so non-Saylor sites can add `chat:mistral` as a third line.

### 7. (Optional) Install the Redash cost-anomaly query (1 min)

If your site uses Redash for cost dashboards and you want a second view independent of Moodle cron, paste the SQL from `.drafts/sola-redash-cost-anomaly-2026-06-09.md` into Redash. Schedule daily; alert on `status = 'ANOMALY'` to the same recipients as `spend_notify_emails`. The Redash query and the v6.0 scheduled task use the same algorithm (today vs rolling 7-day median × MULTIPLIER) and the same rate-card source; running both is a belt-and-suspenders setup that catches the rare case where Moodle cron stops running but Redash is still polling.

### 8. Q&A (rolling)

- **"Will enabling the detector affect anything else?"** No. It's a pure read-only daily query against `local_ai_course_assistant_msgs` × the rate card. Doesn't change config, doesn't touch chat, doesn't send any email except the anomaly alert.
- **"What if Moodle cron is broken on my site?"** The scheduled task won't fire. The Redash query (step 7) is a useful second leg for exactly this case.
- **"Can I tune the alert threshold per course?"** No — the v6.0 detector is site-wide. Per-course tuning is what the `spend_cap_per_course_default` cap (step 3) is for. The detector catches the SITE-WIDE pattern; the cap catches the PER-COURSE pattern. They complement, not replace.
- **"How do I unsubscribe a recipient?"** Same as the existing spend-cap alerts: per-recipient opt-out via `email_optout` (TYPE_SPEND_ALERT). The detector calls the same opt-out check as `spend_guard::maybe_notify()`.
- **"What's the false-positive rate at multiplier=2.0?"** Depends on your normal day-to-day variation. At Saylor's pilot volume (low-thousands of active learners), most days are within ±30% of the median. A 2x ratio is rare without a real cause. Bump to 1.5 only if you want EARLY warnings and don't mind ~1 false positive per month per typical course.
