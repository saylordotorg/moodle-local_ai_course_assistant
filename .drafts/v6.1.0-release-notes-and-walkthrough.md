# SOLA v6.1.0 — Release Notes, Feature Summary, and Admin Walkthrough

## Part 1: Release Notes (10 June 2026)

Admin-UX release from the second half of the five-agent review (the security half shipped as v6.0.1). Theme: make what SOLA already does **observable and operable from the web UI** — cache hits visible in token analytics, premium-routing decisions visible in the category breakdown, settings grouped where admins expect them, and a web emergency panel so incident response no longer requires SSH. One schema column; everything else is settings-page restructuring and analytics surfacing.

- **New: web emergency panel (`emergency_admin.php`).** Per-subsystem kill switches (Chat / Voice / RAG / Outreach / MASTER KILL) mirroring the `emergency_disable.php` CLI flags, with current-state badges, a confirm checkbox, an optional audit-log reason, and one-click restore. Reuses the `emergency_control` class that has powered the CLI since v5.4.5 (including the v5.13 `--chat` real-fix), so the web and CLI paths cannot drift. Linked in red from the settings-page quicklinks bar — the review found incident response was SSH-only; now it's two clicks from Moodle admin home. Audit rows record `web:<username>` as the invoker.
- **New: prompt-cache visibility, end to end.** v5.11.0 captured OpenAI's `prompt_tokens_details.cached_tokens` and Anthropic's `cache_read_input_tokens` in the provider layer but never persisted them — the feature was half-shipped and unobservable. v6.1.0 adds a nullable `cached_tokens` column to the msgs table (upgrade hook `2026061001`), normalizes both vendors' counters into it at write time in `sse.php`, and surfaces a "Cached Tokens (N% of prompt)" card on the token-analytics page. Null-vs-zero is preserved so the hit-rate denominator only counts calls that actually reported cache data.
- **Fix: the token-analytics category breakdown was silently excluding all RAG cost rows.** The by-category query filtered `role='assistant'`, but embedding/rerank cost-log rows are written with `role='system'` — so the "RAG" category has been empty since it was introduced. The query now includes system cost rows; the rerank cost-log (`interaction_type='rerank'`, v5.11) joins the RAG bucket, and the v5.12 premium-router decision rows surface as a "Premium routing (escalation decisions)" category whose response-count column is the number of escalations in the window (the escalated calls' actual Opus tokens remain in the Chat category under the per-model table, which is correct — they ARE chat).
- **Settings page regrouping** (from the information-architecture review of all 227 settings / 31 headings):
  - The five `rerank_*` settings get their own "RAG: Two-stage retrieval (re-ranking)" heading — they were orphaned after `rag_chunksize` with no group boundary.
  - The four `mastery_classifier_*` settings (provider / model / weight / threshold) are now registered together; previously the premium-escalation block split `provider` from its three siblings, so admins tuning the classifier scrolled through an unrelated feature.
  - The premium-escalation heading now follows the complete mastery-classifier group.
- **a11y harness:** `emergency_admin.php` added to the Puppeteer axe-core page list (now 20 pages).

### Review findings verified as already fine (no change needed)

- The per-course "Token Usage & Cost →" cross-link from analytics.php already exists, course-scoped, in the time-range row.
- SSRF guards confirmed on both Voyage providers; no API-key leakage in any new logging; all new SQL parameter-bound; CLIs all define `CLI_SCRIPT`.

### Deferred (logged for future releases)

- Splitting the 227-setting page into multiple admin pages (Providers / Features / Cost & Safety / Advanced) — the highest-leverage UX change but a structural refactor deserving its own release and translations.
- Tabbed layout for the 12-card per-course analytics view.
- Course-scoped Learning Radar chips when a course is selected.
- Voyage provider unit tests with mocked HTTP (the providers are SSRF-guarded and exercised on dev; mocked-response tests remain worthwhile).
- Regex-DoS guard for admin-supplied premium-router triggers (admin-only input; documented risk).

### Backwards compatibility

The schema change is one nullable column — rows written before v6.1.0 have `cached_tokens = NULL` and are excluded from hit-rate math by design. The settings regrouping changes display order only; no setting keys changed, no values are migrated, nothing re-prompts. The category-breakdown fix changes what the token-analytics page DISPLAYS (RAG and premium-routing rows appear for the first time) but not any stored data.

### Testing

- Validators 36/36, jailbreak 32/32.
- PHPUnit full plugin suite: 459/459 (1 pre-existing skip) including 4 new `cached_tokens_persistence_test` tests (assistant-row storage, null-when-unreported, never-on-user-rows, zero-vs-null distinction).
- Puppeteer axe-core a11y: **20/20** pages WCAG2AA (including the new emergency panel).
- Consent-gate check: pass. Behat: 20/20 scenarios, 207/207 steps.
- Upgrade hook `2026061001` verified on local Moodle (adds `cached_tokens` column, idempotent via `field_exists`).
- i18n: 27 new English strings (emergency panel ×26, rerank heading ×2, minus shared); 45-language batch deferred to the next lp-i18n cycle alongside the v5.11–v6.0 backlog.

Plugin version `2026061001`, release `6.1.0`.

## Part 3: Admin Walkthrough (~8 min)

### 0. Setup (1 min)

Upgrade to v6.1.0 and run `php admin/cli/upgrade.php`. Confirm hook `2026061001` succeeds (adds the `cached_tokens` column).

### 1. Find the emergency panel (1 min)

Open SOLA settings. In the quicklinks bar at the top, the red **Emergency controls** link opens `emergency_admin.php`. Five cards — Chat, Voice, RAG, Outreach, MASTER KILL — each showing an Active/DISABLED badge. This is the web equivalent of `admin/cli/emergency_disable.php`; both paths share the same `emergency_control` class and write the same audit rows.

### 2. Smoke the kill switch round-trip (2 min)

On the **Chat** card: type a reason ("post-upgrade smoke"), tick the confirm checkbox, click **Disable**. The badge flips to DISABLED and a notification lists the config keys touched (`emergency_chat_disabled (set to 1)` among them). Open any SOLA course — the drawer shows the "SOLA paused" message. Back on the panel, click **Restore**. Chat resumes. Both actions are in the audit log with `web:<your-username>` as the invoker.

### 3. See the new analytics (2 min)

Open **Token Usage & Cost**. Three things are new:

- A **Cached Tokens** card (appears once post-v6.1.0 traffic exists) showing the absolute count and percentage of prompt tokens served from provider caches — this is the observable form of the OpenAI 50% / Anthropic 90% cache discounts.
- The **By category** table now shows **RAG** rows (embedding + rerank cost logs — these were silently excluded before) and a **Premium routing (escalation decisions)** row whose count is the number of v5.12 escalations in the window.
- The per-course time-range row on the analytics dashboard already links to course-scoped token cost (verified, unchanged).

### 4. Notice the settings regrouping (1 min)

In settings: the rerank block now sits under its own "RAG: Two-stage retrieval (re-ranking)" heading; the four mastery-classifier settings are contiguous; premium escalation follows them as its own complete group.

### 5. Q&A (rolling)

- **"Does the emergency panel bypass the CLI?"** No — same class, same audit trail. The CLI remains the right tool when Moodle itself is unresponsive.
- **"Why is my Cached Tokens card missing?"** It appears once at least one post-v6.1.0 call reports cache data. Pre-v6.1.0 rows are NULL by design.
- **"Did my category totals change?"** The RAG and premium-routing rows are new (previously hidden). Chat numbers are unchanged.
- **"Can a non-admin reach the emergency panel?"** No — `moodle/site:config` capability plus sesskey on every action.

## Part 4: Admin-facing blurb

> **v6.1.0 makes SOLA operable and observable from the web UI.** A new emergency panel puts per-subsystem kill switches (chat, voice, RAG, outreach, master) two clicks from Moodle admin home — incident response no longer requires SSH. Token analytics now shows your prompt-cache hit rate (the observable form of the OpenAI/Anthropic cache discounts), surfaces RAG costs that were silently hidden by a filter bug, and counts premium-routing escalations per window. Settings are regrouped so related options sit together. One nullable schema column; no behaviour change for learners.
