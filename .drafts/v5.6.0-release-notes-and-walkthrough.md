# SOLA v5.6.0 — Release Notes, Feature Summary, and 20-minute Admin Walkthrough

## Part 1: Release Notes (21 May 2026)

<!-- Write a single-sentence headline describing the theme of this release, then
     a bullet list of the user-visible changes. Mirror the style of the
     previous release file in `.drafts/`. -->

User-facing behavior change: every install upgrading to v5.6.0 sees more current-page text in the prompt on page-specific questions and more course content on course-level questions. The structured prompt assembly stops capping each section at a static character ceiling and starts allocating each section's share of the total budget by admin-tunable percentages, with an automatic context-aware boost that shifts allocation toward whichever bucket the learner's question is most likely grounded in.

- **Proportional budget model.** The 12,000-char system prompt now allocates across four buckets (safety_identity / course_structure / course_content / current_page) by percentages summing to 100. Defaults are 10 / 10 / 40 / 40, empirically tuned by a 500-call benchmark on dev that scored 5 candidate weight sets against the 50-prompt golden tutor set. Winning set beat the original memo baseline by 3% on rubric mean.
- **Automatic page-focus boost.** Shifts 15 weight points toward `current_page` when a specific page is in scope (pageid > 0 OR coach mode), or toward `course_content` when no page is selected. Default boost mode is `page_focus` (on by default).
- **Per-coach-mode override.** Optional JSON setting applies only during graded-quiz coach mode. Lets institutions force heavier current_page allocation during quizzes.
- **Admin UI.** Three new knobs in AI Provider & Models: `prompt_section_weights` (JSON), `prompt_context_boost_mode` (off / page_focus / aggressive), `prompt_section_weights_coach` (JSON override).
- **New public API.** `context_builder::section_budgets($total_budget, $pageid, $quizmode)` returns the per-bucket allocation map for any input. `section::max_chars` field added; assembler honors it before drop-on-priority safety net.

### Backwards compatibility

Existing installs that don't configure the new settings see the new empirically-tuned defaults (10/10/40/40 + page_focus boost). Admins who want the prior even-allocation behavior can set `prompt_context_boost_mode=off` and `prompt_section_weights={"safety_identity":25,...}`. `prompt_budget_chars`, `current_page_content_maxchars`, and `maxhistory` are unchanged.

### Testing

- New `tests/section_budgets_test.php`: 8 cases covering all the proportional-allocation paths and the max_chars enforcement.
- Validator suite 36/36 pass.
- Jailbreak suite 32/32 pass (safety_identity 10% floor preserves jailbreak-resistance markers in full).
- Live benchmark on dev.sylr.org against SOLATEST: 500 calls across 5 weight sets, 12.58¢ total spend. Winning set baked in as default.

Plugin version `2026052120`, release `5.6.0`.

## Part 3: 20-minute Admin Walkthrough

This walkthrough is the demo script for showing v5.6.0 to instructors, ops leadership, or external collaborators. It covers proportional prompt allocation end to end. v4.12.0 introduced the structural separation (categories, priorities, drop-on-priority). v5.6.0 finishes the arc by changing *how* the budget is divided up: from static per-section caps to admin-tunable percentage allocations across four high-impact buckets, with an automatic boost that follows what the learner is actually looking at.

### 0. Setup (1 min)

Sign into a course as an admin. Open SOLA via the side tab. Confirm the widget loads and a chat is possible. Have the prompt-debug log ready to tail in a second terminal: `tail -f ~/Sites/moodledata/temp/sola_prompt_debug.log`.

### 1. The "before" picture (2 min)

Frame the problem: v4.12.0 gave us a structured prompt with categories and priorities, but per-section budgets were static character ceilings — `current_page_content_maxchars=2000`, `faq_maxchars=400`, and so on. Two consequences:

- Static ceilings don't know what the learner is looking at. A student on a specific course page gets the same allocation as a student on the course main view, even though page-specific questions need much more current-page text.
- Static ceilings are hard to tune across institutions. Different course shapes (short pages vs long pages, RAG-heavy vs RAG-off) want different splits and there was no single knob.

v5.6.0 collapses ~10 per-section character settings into four percentage allocations.

### 2. Turn on the prompt-debug log (2 min)

Site administration → Plugins → Local plugins → AI Course Assistant → main settings page. Tick **"Log assembled system prompt to file"** (`prompt_debug_enabled`). Save.

Send one chat turn from a student account in the demo course. Tail the log and read out the new per-bucket breakdown:

```
=== 2026-05-21 12:00:00 courseid=2 userid=2 pageid=47 provider=openai ===
Total: 11932 chars (~2983 tokens)
Budget: 12000 chars
Boost: page_focus (pageid=47 → +15 to current_page)
Per-bucket allocation:
[safety_identity ] weight 10  budget  1200  used  1183
[course_structure] weight  5  budget   600  used   598
[course_content  ] weight 30  budget  3600  used  3580
[current_page    ] weight 55  budget  6600  used  6571
```

Point out: each bucket has an explicit weight, an explicit char budget derived from the weight, and an explicit "used" figure. The boost line names *why* the weights drifted from the configured defaults. This is the measurement that's new in v5.6.0.

### 3. The four buckets (3 min)

Walk through the buckets in order:

- **safety_identity** — anti-injection block, identity, output markers, jailbreak resistance. Has a 10% floor and is exempt from per-bucket caps; always lands in full. Floor is enforced by `parse_section_weights()` and validated by the jailbreak suite (32/32 still pass at 10%).
- **course_structure** — course title, FAQ, learning profile, study plan, mastery state — the lightweight scaffolding.
- **course_content** — the heavy chunk: RAG retrievals or stuffed course content. The bucket that benefits most from extra budget when the learner is asking a course-level question.
- **current_page** — the specific page or activity the learner is viewing. The bucket that benefits most from extra budget when a specific page is in scope.

`current_page + course_content` has a combined 40% floor so the model always has substantive material to ground in.

### 4. The context-aware boost (3 min)

Open Settings → **Prompt section proportions** → **Context boost mode**. Three options:

- **off** — base weights apply on every turn regardless of page context.
- **page_focus** (default) — shifts ~15 weight points toward `current_page` when `pageid > 0` OR `quizmode='coach'`; when no page is in scope, redistributes `current_page`'s allocation to `course_content` and `course_structure`.
- **aggressive** — same logic, ~25 points instead of 15. Best when learners consistently ask page-specific questions.

Demo: open the chat from a course page (`pageid > 0`), send a turn, show the log line `Boost: page_focus (pageid=47 → +15 to current_page)`. Then open the chat from the course main view (`pageid = 0`), send a turn, show `Boost: page_focus (no page → +15 redistributed to course_content + course_structure)`. Same admin settings, two different allocations driven by what the learner is looking at.

### 5. Live weight customization (3 min)

Settings → **Prompt section proportions** → **Base section weights (JSON)**. Default is blank (uses benchmarked 10/10/40/40).

Paste a non-default config to show validation behavior:

```json
{"safety_identity": 15, "course_structure": 15, "course_content": 30, "current_page": 40}
```

Save. Send another chat turn from the student account. The log now shows weights 15/15/30/40 (plus the +15 boost on top of `current_page` → effectively 15/15/22/48 after redistribution). Point out: the configured weights are the *base*; boost mode is layered on top per turn.

Now demo the fallback: paste a deliberately invalid config (weights summing to 200, or `safety_identity: 0` below the floor). Save. The log shows the benchmarked defaults reappear — invalid input falls back silently, and `prompt_debug_enabled` is the way to confirm what actually ran. Restore to blank when done.

### 6. Coach-mode override (2 min)

Settings → **Coach-mode override (JSON, optional)**. The graded-quiz coach flow lives in its own mode (`quizmode='coach'`); during a coach turn, this override (if set) replaces the base weights entirely, then the boost is applied on top. Useful when an institution wants quiz coaching to force a heavier `current_page` allocation than chat does.

Paste an override skewed hard toward `current_page`:

```json
{"safety_identity": 10, "course_structure": 10, "course_content": 20, "current_page": 60}
```

Save. Open a graded quiz in the demo course, trigger the SOLA coach panel, send a turn. The log shows the override applied. Send a normal (non-quiz) chat turn from the same course; base weights apply, not the override. Restore to blank when done.

### 7. Cost framing (2 min)

The default 12,000-char budget (`prompt_budget_chars`, unchanged from v5.5.x) caps the prompt at ~3,000 tokens. v5.6.0 doesn't change that ceiling — what it changes is *which* sections get to spend it. Empirically, the 10/10/40/40 + page_focus default beat the original memo baseline (15/15/30/40) by 3% on the 50-prompt rubric mean in the dev benchmark (`admin/cli/run_weight_benchmark.php`, 500 calls, 12.58 cents on gpt-4o-mini).

Pull up Token Usage analytics. There's no meaningful per-turn cost change from v5.5.x to v5.6.0 — total budget is the same. The win is response quality at constant cost.

For institutions wanting to validate against their own course mix, the benchmark CLI is shipped: `sudo -u www-data php admin/cli/run_weight_benchmark.php --courseid=<id>`. Two runs against the same course converged on the same winner during dev tuning. ~12 cents per run on gpt-4o-mini.

### 8. The deferred work (1 min)

Explicitly call out what's NOT done in v5.6.0:

- A per-course override for base weights (today: site-wide setting, plus the coach-mode override). Some courses with unusually long pages may want a different default; deferred to v5.7.x if pilot feedback asks.
- A graphical analytics surface that plots per-bucket char averages over time. Today the data is in the prompt-debug log; surfacing it as a chart on the Token Usage dashboard is a follow-up.
- Auto-tuning of base weights from observed traffic. The benchmark CLI is the manual version. Auto-tuning at site-level granularity would need a feedback loop on rubric judges, which is a v6 conversation.

### 9. Q&A (rolling)

Common questions and tight answers:

- **"Will this break existing installs?"** No. New installs and upgrades both get the benchmarked default (10/10/40/40 + page_focus) which empirically improves response quality at constant cost. Admins who want exact prior even-allocation behavior can set boost_mode=off and weights={"safety_identity":25,"course_structure":25,"course_content":25,"current_page":25}.
- **"What about jailbreak resistance? You're only giving safety 10%."** The 10% floor was chosen precisely so the safety_identity block fits in full at the default 12,000-char budget. Jailbreak suite passes 32/32 at this floor. If you lower `prompt_budget_chars` below ~8,000, raise the safety floor by setting the JSON manually.
- **"How do I know the right weights for my institution?"** Run the benchmark CLI against a representative course. Two runs converging on the same winner is the signal you've found a stable optimum. Don't change the baked-in default without doing this; the 10/10/40/40 is tuned against SOLATEST, which may not look like your course shape.
- **"What's the difference between this and the old per-section character settings?"** The old static caps still exist for the long tail of small sections (`current_page_content_maxchars`, etc.) but are now subordinate to the per-bucket budgets. If a section's per-bucket allocation is smaller than its static cap, the per-bucket allocation wins. If a section's static cap is smaller, that wins. Effectively: per-bucket budgets are the ceiling, static caps are the floor.
- **"Does this affect the learner experience?"** Yes, in the intended direction: page-specific questions get more current-page text; course-level questions get more course content. Same model, same total prompt size, different allocation. Learners shouldn't see anything visually new; just more relevant grounding in answers.
