# SOLA — Feature List, Release Notes (v5.4.5 → current), and Demo Scripts

Compiled from the shipped changelog. Release notes run v5.4.5 (the production-pilot
hardening baseline) through the current head. The demo scripts are **version-free**
(no release numbers, so they don't go stale) and **word-for-word** (read the quoted
lines aloud; do the [bracketed] actions).

# 1. Full Feature List (cumulative)

## The learner experience
- AI learning coach embedded in course pages, opened from a side tab that expands into a chat drawer (or a floating button at the corner). Draggable, resizable, 2-state expand, size remembered.
- Multi-provider AI backend with live token streaming (OpenAI, Claude, Mistral, Llama via Together, xAI, DeepSeek, Gemini, and self-hosted/Ollama-compatible).
- Conversation starters on open, dynamic next-step suggestion chips after each answer, a 50-exchange conversation cap, and per-message actions (copy, bookmark, rate).
- First-run consent notice with a scroll-to-agree gate, shown once per learner (GDPR Article 13 disclosure).

## Course-aware grounding
- Structured prompt builder that assembles the system prompt from prioritized sections with a drop-on-priority safety net.
- Proportional prompt budgeting across four buckets (safety/identity, course structure, course content, current page) with admin-tunable weights and an automatic context-aware boost toward whatever the learner's question is grounded in.
- Retrieval-augmented search over course content (embedding-based), with drift auto-reindex and PDF text extraction.
- Current-page awareness; truncation-aware "I couldn't search the whole course" answers instead of false "not found" claims.

## Pedagogy
- Practice quizzes (setup, interactive cards, score summary), study plans, scheduled reminders.
- Mastery tracking against per-course objectives; a mastery-aware "what should I focus on?" starter that names the learner's weakest objective.
- Cross-course mastery rollup: recognizes a competency already demonstrated in another course and builds on it instead of re-drilling.
- Forward learning-path awareness: tells the learner where the current course leads next in their program (Degrees and Learn), with concept bridges.
- Visual learning-path panel ("my path"): the program as course nodes (done / current / upcoming), each expandable to objectives and mastery.
- Next-course readiness nudge: a gentle banner + conversational line when the learner is ready to move on.
- Pedagogy modes: Socratic guidance, worked examples, flashcards, a Python code sandbox, essay feedback — each a per-course toggle.
- Personalized greeting, nicknames, encouragement style; a per-learner learning profile.

## Languages
- 46-language interface with automatic language detection; course help and the UI work in the learner's language.

## Future direction (not in the current plan)
- Spoken voice interaction (speech-to-text input, text-to-speech read-back, a realtime voice mode) and on-screen talking avatars exist in the codebase but are **not** part of the current text-based rollout. Treat as roadmap only; do not present them as current capabilities.

## Analytics and operations
- Usage analytics and provider-comparison dashboards.
- Learning Radar: multi-schedule reporting, Redash export, citations, anomaly digest.
- Golden tutor benchmark harness (50-prompt rubric set; cost-vs-quality Pareto report).
- Per-call provider failover with a circuit breaker; spend caps; LLM rate-card auto-refresh; vendor DPA gating.

## Self-hosted readiness
- Token-aware prompt budgeting for small-context backends (declare the backend window; the prompt is sized to fit).
- Bounded pre-first-byte retry on transient 429/503; deployment presets; an on-demand backend self-test; deployment-sizing guidance.

## Safety, privacy, security
- Jailbreak resistance (a 32-case corpus run every release), runtime validators, an off-topic cap, and moderation.
- Privacy provider with GDPR erasure across all stored data, including email opt-outs; audit logging that records lengths, not message text.
- Consent gate; disclosed consent before any support-desk escalation.
- Hardened against SSRF, XSS, injection, and cross-course access; documented provider-key scoping; an emergency disable control (whole plugin or per subsystem) with an audit trail.

## Administration
- Per-course configuration; pedagogy defaults resolvable site-wide or per course; branding/UI controls; a navigable settings page.

# 2. Release Notes (newest first, back to v5.4.5)

**v5.10.2 — Consent scroll-gate fix.** The first-run consent gate read the notice's height at page load while the drawer was still closed (zero height), so "scrolled to bottom" was trivially true and Accept enabled before the learner saw the notice. Now the check waits until the region has real layout and re-runs on drawer open. Adds a Puppeteer regression harness. One-template change; no settings or schema change.

**v5.10.1 — Security-audit remediation, consent scroll-gate, full i18n.** Remediates an internal security/PII audit: privacy erasure now covers the email opt-out table; a media-viewer endpoint now verifies session ownership (closes an IDOR); support-desk escalation now requires disclosed consent (new toggle, default on); de-anonymized analytics exports are audit-logged. Adds the first-run consent scroll-to-agree notice and translates all new strings into 45 languages. Suite 405/405.

**v5.10.0 — Self-hosted readiness: token-aware budgeting, backend retry, sizing tools.** A new "backend context window" setting sizes the prompt to fit a small self-hosted model's window (token-aware, per-language). Bounded pre-first-byte retry on 429/503. A deployment-presets page, a live backend self-test page, the metrics tuner capped at the window, truncation-aware not-found answers, and a deployment-sizing wiki page. Default-off; hosted deployments unaffected.

**v5.9.0 — Learning-path map + next-course nudge.** A "my path" panel renders the learner's program as expandable course nodes (status, objectives, mastery); a readiness nudge surfaces a banner + conversational line when the learner completes the course or masters ~80% of its objectives. One aggregator behind a panel web service, the banner, and the prompt line. Off by default; learner's own allocation only; WCAG-compliant; 19 strings in 45 languages.

**v5.8.1 — Accessibility fix + test-harness cleanup.** Underlines links in muted helper text on the plugin's admin pages so they aren't color-only (WCAG 1.4.1); the a11y harness goes from 3 violations to 0. Retires the dead pa11y-ci fallback; the axe-core runner is now self-contained.

**v5.8.0 — Forward learning-path awareness.** A conversational block tells the learner where their course leads next in their program (Degrees and Learn), with the next course and a concept bridge where the data supports it. Asserts a specific next course only where the program encodes one; any-order programs degrade to position-only. Reads the learner's own allocation; off by default; no schema change.

**v5.7.0 — Cross-course mastery rollup + mastery-aware starter.** Recognizes a competency a learner already demonstrated in another course (matched by competency reference or fuzzy objective-title similarity, precomputed nightly) and acknowledges it instead of re-teaching. Personalizes the "what should I focus on?" starter to the learner's weakest objective. One additive table; both features default off.

**v5.6.1 — CI fix.** Restores green CI after v5.6.0: fixes a cache-key that rejected hyphenated provider labels, a truncation-reporting bug, a forbidden falsy-default pattern, and PHPDoc type annotations the Moodle checker rejects.

**v5.6.0 — Proportional prompt allocation + context-aware boost.** The system-prompt budget is allocated across four buckets by tunable percentage weights (empirically tuned defaults 10/10/40/40) instead of static per-section caps, with an automatic boost toward the current page on page-specific questions and toward course content otherwise. A user-visible improvement on grounding; admins can restore the old even allocation.

**v5.5.6 — Admin UX polish.** Renames the widget gear tooltip to "Course settings," makes per-message action icons always visible on the latest reply (and on keyboard focus / touch), reorders the settings table of contents, and adds a "Save" jump link.

**v5.5.5 — Follow-up to v5.5.4.** Corrects role resolution so site admins resolve as staff (not student) and don't trigger an unnecessary profile-generation LLM call on every admin chat.

**v5.5.4 — Security hardening.** Eleven fixes from a five-scan red-team audit: closes a cross-course coach-mode bypass (High), scrubs SSE control markers per chunk, course-scopes page lookups, sanitizes PDF filenames, blocks redirect-based SSRF in the transcript fetcher, caps the SSE client buffer, and hardens several UI innerHTML paths. Jailbreak 32/32 maintained.

**v5.5.3 — Styling fix.** Model names in the rate-card view use a muted blue color instead of a monospace font (code editors keep monospace). CSS only.

**v5.5.2 — Comparison-provider base URL.** Adds an optional base-URL column to comparison providers so one provider class can route through alternate or self-hosted OpenAI-compatible endpoints. Backward compatible.

**v5.5.1 — Golden tutor benchmark harness.** A CLI that runs 50 golden prompts through every configured provider, scores them with a rubric LLM on Socratic guidance / accuracy / tone, and reports the cost-vs-quality Pareto frontier with a decision rule. Purely additive.

**v5.5.0 — Per-call failover.** A circuit-breaker decorator rotates providers on per-call timeout or 5xx (not just on a spend cap), with audit rows as SOC 2 evidence. Streaming only falls over before the first token. Off by default.

**v5.4.6 — RAG latency instrumentation.** Adds a per-message `rag_latency_ms` column and the timing code, for capacity-planning monitoring.

**v5.4.5 — Production security hardening (pilot baseline).** The production-readiness release for the 30-course pilot: enables Dependabot scanning, confirms a clean secrets sweep, documents per-provider key scoping and rotation, audits logging (no message content logged), and adds an emergency-disable mechanism (whole plugin or per subsystem) with a runbook and audit trail.

## 3a. Student Overview — 5 minutes

**[BEFORE YOU START]** [Sign in as a demo student enrolled in a course that has the assistant enabled, ideally a course that belongs to a program. Be on the course page with the assistant not yet opened.]

"This is your course assistant. It lives on the right edge of every course page, and it's there to help you while you're studying, without you ever leaving the page you're on. Let me show you what it does.

[Click the side tab to open the assistant.]

The first time you open it, you'll see a short notice about how it works and what it does with your information. You read down to the bottom, and then you can accept. After that first time, you won't see it again.

[Scroll the notice, click Accept.]

Now it's open. Notice it already suggests a few things you might ask, so you're never staring at a blank box. I'll ask it something about this course.

[Type a real course question, e.g. 'Can you explain the main idea of this section?' Send it.]

A couple of things to notice while it answers. It's streaming the reply as it writes, so you're not waiting on a blank screen. And it actually knows this course. It's not a generic chatbot, it's reading the material for the course and the page you're on, so its answer is about what you're actually studying. If you ask it about something in a specific page, it'll focus on that page.

[When the answer finishes, point at the suggestion chips.]

After it answers, it offers a few natural next steps, so you can keep going with one tap instead of typing.

Two of my favorite things. First, you can ask it to quiz you. [Use the 'practice quiz' starter or ask 'quiz me on this section'.] It'll make practice questions, you answer them, and it shows you how you did. It's a way to check yourself before a real assessment.

Second, you can ask it what to focus on. [Use the 'what should I focus on?' starter.] If your course tracks learning goals, it will point you at the specific thing you're weakest on, instead of a vague 'study more.'

If your course is part of a degree or a certificate, there's a 'my path' button at the top. [Open the path panel.] This shows you your whole program as a sequence of courses, where you are in it, what you've finished, and what's coming next. You can open any course to see its goals and how you're doing on each. And when you've got the current course handled, it'll gently let you know you're ready for the next one.

A few more things quickly. It works in dozens of languages, so you can use it in the language you're most comfortable in. You can bookmark a helpful answer, copy it, or give it a thumbs up or down.

That's the whole idea. It's a patient tutor that knows your course, quizzes you, points you at what matters, and shows you where you're headed, right there on the page while you study. Open it whenever you're stuck."

This is what a learner sees: a side tab that opens a course-aware chat coach. It reads the course content, so its help is grounded in the actual material, not generic.

## 3c. Admin Walkthrough — 20 minutes (full)

**[BEFORE YOU START]** [Admin login. Open: a course page with the assistant; the plugin settings page; the per-course settings page; the analytics view. Have the prompt-debug log handy if you want to show grounding.]

### 1. What it is (1 min)
"This is an AI learning coach embedded in Moodle course pages. A learner opens it from a side tab and gets a tutor that's aware of the specific course and page they're on. I'm going to walk the whole administration surface: how it connects to a provider, how you keep cost and safety under control, the teaching behaviors you can switch on, and the privacy and operations story. Stop me with questions as we go."

### 2. The learner's view (2 min)
[Open the assistant on the course page.]
"Here's the experience we're configuring toward. A side tab opens the chat. It greets the learner, offers starter prompts, and answers questions about this course, grounded in the actual course content, streaming as it goes. After each answer it suggests next steps. Learners can bookmark answers and use it in their own language. Everything from here on is about shaping this safely and cost-effectively."

### 3. Provider, models, failover, spend (3 min)
[Go to plugin settings → AI provider & models.]
"First, the engine. You pick a provider and model, the plugin supports the major hosted providers and self-hosted OpenAI-compatible backends. Two things worth setting up properly.

One: failover. You can define an ordered chain, a primary plus fallbacks. If a call times out or the provider returns a server error, a circuit breaker rotates to the next provider and opens a short cooldown on the failing one. Every fall-through writes an audit row, which is your evidence of resilience for a SOC 2 review. It's off by default; turn it on once your legal team has the data-processing terms in place for each provider in the chain.

Two: spend. You can set monthly caps, and the rate card for model pricing refreshes automatically, so your cost math stays current."

### 4. Grounding: prompt budgeting and retrieval (3 min)
"Now, why its answers are about your course and not generic.

The assistant builds its instructions from prioritized sections, and it allocates a fixed character budget across four buckets: its own safety and identity, the course structure, the course content, and the current page. You can tune those proportions. And it boosts automatically, when a learner asks about a specific page, it shifts the budget toward that page's text; when they ask a broad course question, it shifts toward course content. The defaults are tuned from a benchmark, so most sites never touch this.

On top of that is retrieval: the assistant searches your course material, including text pulled from PDFs, to find the most relevant passages for each question. If it can't fit everything and the learner isn't on a specific page, it's honest about it, it says it couldn't search the whole course and suggests opening the relevant page, rather than wrongly claiming the topic isn't there."

### 5. Pedagogy defaults (4 min)
[Go to pedagogy defaults.]
"This is the teaching layer, and it's where you'll spend the most time. Every one of these is off by default and resolvable either site-wide or per course, so you can pilot on one course first.

- Mastery tracking watches how a learner is doing against course objectives, and a 'what should I focus on?' starter points them at their weakest one.
- Cross-course mastery means that if a learner already proved a skill in another course, the assistant acknowledges it instead of re-drilling it.
- The learning-path features are the program view: a conversational hint about where the course leads next, a visual panel of the whole program, and a gentle nudge when the learner is ready for the next course. These read the learner's own program enrolment only, and they stay silent for courses that aren't in a program.
- And there are teaching modes, Socratic questioning, worked examples, flashcards, a code sandbox for programming courses, and essay feedback.

The design principle across all of these: advisory, learner's-own-data-only, and default off, so an upgrade never changes behavior until you choose to switch something on."

### 6. Safety and moderation (2 min)
"Guardrails. The assistant resists jailbreak and prompt-injection attempts, and we run a corpus of those attacks against every release. There's an off-topic cap so it stays on coursework, content moderation, and a runtime validator layer you can enable for stricter enforcement. The safety and identity portion of its instructions is protected, it can't be squeezed out by other content, so its guardrails are always present."

### 7. Privacy and consent (2 min)
"Privacy. The first time a learner opens the assistant, they see a consent notice, a plain-language disclosure of what it does with their information, with a link to the full privacy notice. They have to read to the bottom before they can accept, and they only see it once.

Behind that: full data-erasure support, a right-to-be-forgotten request removes everything we store about a learner, including their email opt-out. Our logs record lengths and events, never message text. And if a conversation ever escalates to your support desk, that sharing is itself disclosed and gated on consent."

### 8. Languages and reach (1 min)
"Reach. The interface and the help work in dozens of languages, detected automatically, which matters for a global, self-paced audience. On the roadmap, but not in the current rollout, are spoken voice interaction and on-screen avatars; I'll flag those as future direction rather than something it does today."

### 9. Analytics and operations (1 min)
[Show analytics.]
"Operations. There's a usage and provider-comparison view, and a reporting layer that can export to your analytics stack, cite sources, and flag anomalies. For tuning, there's a benchmark harness that scores providers on quality versus cost so your model choice is evidence-based rather than a guess."

### 10. Self-hosted and emergency controls (1 min)
"Two last things. If you run your own model on modest hardware, the assistant can size its prompt to fit a small context window, retry transient overloads, and self-test the backend on demand, there are presets and a sizing guide for that. And the safety net: a single command takes the assistant fully offline, or scopes the shutdown to one subsystem, in seconds, with an audit row recording who did it and why. Reverse it just as fast.

So, to sum up: configure the provider and your cost and safety guardrails once, switch on the teaching behaviors you want per course, and you have a course-aware AI coach that's grounded in your material, private by design, observable, and controllable. Where would you like to go deeper?"
