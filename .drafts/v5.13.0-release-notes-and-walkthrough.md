# SOLA v5.13.0 — Release Notes, Feature Summary, and Admin Walkthrough

## Part 1: Release Notes (9 June 2026)

Defensive-defaults release. Three changes: (1) a new `spend_cap_per_course_default` admin setting that propagates a single dollar amount to every course without an explicit override (so a runaway course is bounded even when no one has tuned its individual settings), (2) a production bug fix in `emergency_control --chat` that was silently no-op-ing from v5.4.5 through v5.12.x, and (3) the formal removal of Mistral from Saylor's default failover chain. No schema change.

- **New: `spend_cap_per_course_default` admin setting.** Sets a single dollar amount that `spend_guard::get_cap()` falls back to for any course without its own per-course override. Default is `0` (no default, preserving prior behaviour). Saylor production should set this to `$30` to bound any single course's monthly spend without having to tune individual courses. The existing 80% / 95% / 100% notification email pipeline (`spend_notify_emails` recipient list, falling back to site admins) carries through unchanged.
- **Fix: `emergency_control --chat` was a silent no-op from v5.4.5 through v5.12.x.** It wrote `spend_cap_site=0` thinking 0 = paused, but `spend_guard::get_cap()` treats 0 as unlimited. v5.13 adds a dedicated `emergency_chat_disabled` flag that `spend_guard::check()` consults before the cap math, so `--chat` now actually blocks chat traffic. Round-trip verified end-to-end on local Moodle (disable → check returns `blocked` → restore → check returns `ok`). All 10 existing `emergency_control` PHPUnit tests still pass with the additional flag wired.
- **Saylor defaults: Mistral removed from the failover chain.** Saylor's `spend_failover_chain` on dev is now `chat:openai` only (was `chat:openai\nchat:mistral` in the v5.11 vendor doc spec). Mistral was originally specified as the EU-resident ultimate fallback. Removed because (a) Saylor doesn't need EU residency at current scale, and (b) the Mistral training-opt-out + ZDR procurement work isn't worth doing for a fallback that may never fire. **The Mistral provider class stays available** so non-Saylor sites can still select it; they (not Saylor) are responsible for the procurement steps in `.drafts/sola-v5.11.0-external-actions.md`.
- **6 new PHPUnit tests** (`tests/spend_guard_per_course_default_test.php`) covering the default-fallback path: applies when no explicit override; falls through to site cap when default is 0 or unset; site-wide call ignores the per-course default; capability cap beats per-course default. Plus 1 test marked skipped (probes the legacy per-course-override path, which was dead code in the existing spend_guard).
- **CLAUDE.md refresh** to v5.12.1 current (Saylor production is v5.4.5; dev is v5.12.1 → v5.13.0 with this release). Adds Vendor stack snapshot section. New Key Files entries for `spend_guard`, `emergency_control`, `premium_router`, Voyage providers, conversation_classifier, claude_provider. Upcoming Work list refreshed (Mistral procurement removed from Saylor's list).
- **Vendor recommendation docs updated** to reflect Saylor's Mistral-removed posture: `sola-vendor-recommendations-2026-06-09.md`, `sola-vendor-optimization-by-mau-2026-06-09.md`, `sola-v5.11.0-external-actions.md` all now distinguish Saylor defaults from non-Saylor options.
- **New: `.drafts/sola-redash-cost-anomaly-2026-06-09.md`** — daily Redash query that catches cost spikes the existing cap thresholds miss (runaway course, accidental premium-tier enable, provider misroute). Compares today's spend vs the rolling 7-day median × 2 multiplier; alert pipeline uses the existing `spend_notify_emails` recipient list.

### Backwards compatibility

`spend_cap_per_course_default` defaults to `0` (unset), so the only behaviour change for existing sites is the `emergency_control --chat` fix — which only affects sites that have actually used `--chat` in the past (where the change is "the no-op now works"). All other paths are unchanged.

## Part 3: Admin Walkthrough (~10 min)

Audience: admin standing up the defensive defaults for SOLA on their site after a v5.13.0 upgrade.

### 0. Setup (1 min)

Have admin access to Site administration → Plugins → Local plugins → AI Course Assistant. Confirm spend-notify recipient list under **Spend notifications** is what you want (or leave blank to fall back to site admins).

### 1. Set the default per-course cap (2 min)

Site administration → Plugins → Local plugins → AI Course Assistant → **Spend caps** group. Look for the new **Default per-course spend cap (USD)** field (`spend_cap_per_course_default`). Set it to `30` (or whatever defensive ceiling you want per course per month). Save.

This applies to every course that does not have its own explicit per-course override. The 80% / 95% / 100% email alerts fire automatically when a course crosses each threshold (existing `spend_notify_emails` recipient list, falling back to site admins). A specific course can always raise its own ceiling by setting a higher per-course override.

If you want the default to NOT apply (preserve v5.12.x behaviour exactly), leave it at `0`.

### 2. Verify the emergency kill switch (2 min)

Confirm the `--chat` flag now blocks traffic. From the admin shell:

```bash
php local/ai_course_assistant/admin/cli/emergency_disable.php --chat --reason="post-upgrade smoke"
```

The CLI output should confirm `emergency_chat_disabled (set to 1)`. Hit any SOLA-enabled course's chat drawer; it should show the friendly "SOLA is paused" message instead of a real chat response. Then restore:

```bash
php local/ai_course_assistant/admin/cli/emergency_disable.php --restore --chat
```

Confirm chat resumes. The flag is unset on restore.

If you skipped this step on prior releases (v5.4.5 through v5.12.x), you wouldn't have noticed the silent no-op because the round-trip didn't change behaviour. v5.13 is the first release where `--chat` actually does what its name says.

### 3. Confirm Saylor default failover (1 min)

For Saylor production: the failover chain on dev as of 2026-06-09 is `chat:openai`, with `failover_per_call_enabled = 1`. Mistral is NOT in the chain.

If you're running SOLA at a non-Saylor site and want the Mistral EU-resident ultimate fallback, complete the procurement steps in `.drafts/sola-v5.11.0-external-actions.md` (Mistral training-opt-out + ZDR in their portal), then add `chat:mistral` as a third line in `spend_failover_chain`.

### 4. Optional: install the Redash cost-anomaly query (3 min)

Paste the SQL from `.drafts/sola-redash-cost-anomaly-2026-06-09.md` into a new Redash query. Schedule it daily at 09:00 site-time. Configure an alert: when the `status` column equals `ANOMALY`, send to the same email list as the spend-notify recipients. Catches three failure modes the existing cap thresholds miss: runaway course, accidental premium-tier enable, provider misroute.

Optional companion query (per-course breakdown when an anomaly fires) is in the same draft.

### 5. Q&A (rolling)

- **"Will the default cap break existing courses?"** No — default is 0 (off). You explicitly opt in by setting a dollar value.
- **"What does 'soft cap' mean here?"** The cap blocks new requests for the rest of the period when crossed. SOLA shows the "budget paused" message; learners can still see prior conversation. The cap resets at the start of the next period (default monthly).
- **"Can a course override the default?"** Yes. Per-course `spend_cap_monthly` (via the course-settings UI) beats the global default. The global default is the *minimum* defensive ceiling.
- **"Why was Mistral removed from Saylor defaults?"** Two reasons: Saylor doesn't need EU residency at current scale, and the Mistral training-opt-out + ZDR procurement is real work for a fallback that may never fire. Provider class stays available so we can re-add it later if either changes.
- **"Did the emergency_control fix change anything for the other flags?"** No. `--all`, `--voice`, `--rag`, `--outreach` were already working (they touch real config flags). Only `--chat` had the silent-no-op bug. All four restore paths are also intact.

Companion Redash cost-anomaly query catches the three failure modes the existing cap thresholds miss. No schema change; default `spend_cap_per_course_default` is `0` (off) so existing sites upgrade without behaviour change except the kill-switch fix.
