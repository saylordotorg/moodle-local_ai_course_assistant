## Root cause #2

On first drawer open, `chat.js:4554` calls `UI.preWelcome()` (when the user hasn't dismissed the welcome panel). That helper adds the `.local-ai-course-assistant__drawer--welcome` class to the drawer. `styles.css:3139` hides starters, messages, typing indicator, and input area while that class is on the drawer — to give the welcome panel a clean canvas.

So even with consent dismissed, the first-time drawer open still hides the starters until the user clicks the welcome panel's **Continue** CTA.

## What I'd do differently

The two-overlay pattern (consent + welcome) was implicit in v5.3 work and not documented. v5.3.27 → v5.3.28 → v5.3.29 is three release cycles to land what should have been one tightly-scoped Behat addition. The systemic issue is that I shipped Behat scenarios without local Behat to validate them. **Setting up local Behat is now the highest-value next investment** — would have caught both overlay gates pre-tag.
