# SOLA v6.4.0 — Release Notes, Feature Summary, and Admin Walkthrough

## Part 1: Release Notes (11 June 2026)

Signed policy bundles: change SOLA's behavior in production without a code deploy. A daily task fetches a cryptographically signed JSON file from a URL you control and applies the contained settings — prompts shape, routing, escalation triggers, RAG tuning, spend policy — with a hard allowlist, replay protection, and a full audit trail. Built for sites on managed hosting where every code deploy costs a provider engagement.

- **The mechanism.** A bundle is one JSON envelope: format id, base64 payload, base64 Ed25519 detached signature. The daily `policy_bundle_sync` task (06:20) fetches it from `policy_bundle_url` and applies it only when ALL of the following hold, failing closed otherwise: (1) the signature verifies against the admin configured public key — the private key lives only with the bundle author, so a compromised hosting bucket cannot alter site behavior; (2) every settings key is on the hard coded allowlist (42 behavior settings: routing, prompt budgeting and proportions, history selection, mastery classifier, premium escalation, RAG floor/boost/rerank, spend caps and anomaly policy) — API keys, URLs, webhooks, email recipients, SSRF, consent, and emergency settings can never be set by a bundle, signed or not; (3) the payload version is strictly greater than the last applied version (replay and rollback protection).
- **Audit trail.** Every application writes an audit row with the bundle version, comment, and each changed key with old and new values. The settings page shows last sync time, outcome, and applied version.
- **Authoring tooling.** `admin/cli/policy_bundle_tool.php` does the whole lifecycle: `--keygen` (Ed25519 keypair, private key written mode 0600 with a do-not-upload warning), `--sign` (pre flights the same allowlist and shape checks the receiving site runs, so a bad bundle fails at signing time, not at 06:20 on every site), `--verify`, `--status`, and `--sync` (run a sync now).
- **Data, never code.** No PHP, JS, or templates can travel through a bundle. Code changes still go through git, the test gate, and the deploy pipeline. Off by default; with `policy_bundle_enabled` off the plugin behaves exactly as v6.3.0.
- Companion strategy doc: `.drafts/sola-upgrade-independence-2026-06-11.md` (the three levers for reducing managed hosting engagements, including a ready to send note to Catalyst).

### Backwards compatibility

No schema change. Three new settings (enabled, URL, public key), all empty/off by default. One new scheduled task, inert until enabled.

## Part 3: Admin Walkthrough (~12 min)

### 0. Setup (1 min)

Upgrade to v6.4.0 and run `php admin/cli/upgrade.php`. No schema change. Confirm the new task appears: Site administration, Server, Scheduled tasks, "SOLA signed policy bundle sync" (daily 06:20).

### 1. Generate your keypair (1 min, once ever)

On YOUR machine (not the server):

```
php admin/cli/policy_bundle_tool.php --keygen --out=$HOME/.sola
```

The private key lands in `$HOME/.sola/sola_policy_private.key` (mode 0600). It never leaves your machine: not the repo, not Dropbox, not S3. The command prints the PUBLIC key — copy it.

### 2. Configure the site (1 min)

SOLA settings, "Signed policy bundle (remote behavior updates)":
- **Enable policy bundle sync:** on
- **Policy bundle URL:** where you will host the file, for example `https://your-bucket.s3.amazonaws.com/sola/policy.json`
- **Policy bundle public key:** paste from step 1

### 3. Author and publish a bundle (3 min)

Write `bundle.json`:

```json
{
  "version": 1,
  "issued_at": "2026-06-11T13:00:00Z",
  "comment": "trim rerank pool per the June benchmark",
  "settings": {
    "rerank_candidates": 30,
    "premium_escalation_triggers": "derive|prove that|step by step"
  }
}
```

Sign and upload:

```
php admin/cli/policy_bundle_tool.php --sign --payload=bundle.json --key=$HOME/.sola/sola_policy_private.key --out=signed.json
aws s3 cp signed.json s3://your-bucket/sola/policy.json
```

The `--sign` step refuses anything off the allowlist or malformed, so mistakes fail on your machine, not in production overnight.

### 4. Watch it apply (2 min)

Either wait for 06:20 or force it: `php admin/cli/policy_bundle_tool.php --sync` on the server. Then:
- The settings page heading shows "Last sync: applied version 1 (2 settings changed)".
- The audit log has a `policy_bundle_applied` row listing each changed key with old and new values.
- Run `--sync` again: "skipped: bundle version 1 is not newer than applied 1" — replay protection working.

### 5. See it refuse bad input (2 min)

Edit the hosted JSON by hand (change any byte of the payload) and `--sync`: "Policy bundle rejected: signature verification FAILED", and no setting changed. This is the property that makes remote behavior updates safe: possession of the hosting bucket is not possession of your site's behavior.

### 6. Q&A (rolling)

- **"Can a bundle change API keys or where SOLA sends data?"** No. The allowlist is hard coded in the plugin and excludes every key, URL, webhook, email, and security setting. A bundle containing any such key is rejected in full.
- **"What if the private key is lost?"** Generate a new keypair, paste the new public key into the setting. Old bundles stop verifying, which is the correct outcome.
- **"What if the private key is stolen?"** The thief can change allowlisted behavior settings until you rotate the public key on the site — one settings change. They still cannot touch keys, URLs, or code. Treat the private key like an SSH key.
- **"Does this update code?"** Never. JSON data only, against a fixed allowlist. Code ships through git and the deploy pipeline.
- **"Multiple sites?"** Point them all at the same URL with the same public key — one signed file updates the fleet, each site auditing locally.
