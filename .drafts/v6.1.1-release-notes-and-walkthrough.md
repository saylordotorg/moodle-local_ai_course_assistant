# SOLA v6.1.1 — Release Notes, Feature Summary, and Admin Walkthrough

## Part 1: Release Notes (10 June 2026)

Single-fix patch: the flashcards review page no longer fatals on Moodle 5.x. v6.1.0 is otherwise unchanged. Recommended for any site on Moodle 5.0+ where the flashcards pedagogy feature is reachable. No schema change.

- **Fix: the flashcards review page (`flashcards.php`) fataled on Moodle 5.0+ when flashcards was not enabled for the course.** The not-enabled guard called `print_error('flashcards_disabled', ...)`, but `print_error()` was **removed in Moodle 5.0** — so on the dev fleet's Moodle 5.x sites the page raised "Call to undefined function print_error()" instead of rendering a clean error page. It was the **only** `print_error()` left in the plugin; every sibling page (`essay_feedback.php`, `sandbox.php`) had already migrated to `moodle_exception` during the Moodle-5 compatibility pass. The call also referenced a nonexistent string key (`flashcards_disabled`); the intended `flashcards:disabled` already existed in all 45 translations but was missing from English. The guard now throws `\moodle_exception('flashcards:disabled', ...)` and the English string is present, so the page renders a proper "Flashcards are not enabled for this course" notice. When flashcards **is** enabled, the review flow (due cards, reveal, Again/Hard/Easy self-grade, SM-2 scheduling) is unchanged.

### Backwards compatibility

Two-line change — the `flashcards.php` guard plus one English lang string. No schema change, no settings change, and no behaviour change when flashcards is enabled. Strongly recommended for Moodle 5.x sites because the prior code hard-failed the review page on the not-enabled path.

### Testing

- Validators 36/36; jailbreak 32/32; PHP lint clean on both changed files; all 46 locales carry `flashcards:disabled`.
- Flashcard machinery (save / due / SM-2 review / `get_module_content`) verified on local Moodle 4.5.
- Live dev-runtime verification on **Moodle 5.0 (dev500), 5.1 (dev501), and 5.3dev (dev503)**: the not-enabled path now returns a clean `moodle_exception` with the resolved message (no `print_error` fatal); the enabled path saves a card, lists it as due, renders it, and records a review.
- HTTP boot check on dev500: `flashcards.php` returns a `303` login redirect (no parse/boot fatal).

Plugin version `2026061002`, release `6.1.1`.

## Part 3: Admin Walkthrough (~8 min)

> v6.1.1 is a patch over v6.1.0 — the admin surface is unchanged. The walkthrough below is carried forward; the only behavioural change is that the flashcards review page (`flashcards.php`) now renders a clean "not enabled" notice on Moodle 5.x instead of a fatal.

### 0. Setup (1 min)

Upgrade to v6.1.1 and run `php admin/cli/upgrade.php`. No schema change in this patch; the prior `2026061001` hook (the `cached_tokens` column from v6.1.0) remains in place.

### 1. Find the emergency panel (1 min)

Open SOLA settings. In the quicklinks bar at the top, the red **Emergency controls** link opens `emergency_admin.php`. Five cards — Chat, Voice, RAG, Outreach, MASTER KILL — each showing an Active/DISABLED badge. This is the web equivalent of `admin/cli/emergency_disable.php`; both paths share the same `emergency_control` class and write the same audit rows.

### 2. Smoke the kill switch round-trip (2 min)

On the **Chat** card: type a reason ("post-upgrade smoke"), tick the confirm checkbox, click **Disable**. The badge flips to DISABLED and a notification lists the config keys touched (`emergency_chat_disabled (set to 1)` among them). Open any SOLA course — the drawer shows the "SOLA paused" message. Back on the panel, click **Restore**. Chat resumes. Both actions are in the audit log with `web:<your-username>` as the invoker.

### 3. See the analytics (2 min)

Open **Token Usage & Cost**. Note the **Cached Tokens** card (absolute count and percentage of prompt tokens served from provider caches — the observable form of the OpenAI 50% / Anthropic 90% cache discounts), the **By category** table showing **RAG** rows and a **Premium routing (escalation decisions)** row, and the per-course time-range cross-link to course-scoped token cost.

### 4. Notice the settings grouping (1 min)

In settings: the rerank block sits under its own "RAG: Two-stage retrieval (re-ranking)" heading; the four mastery-classifier settings are contiguous; premium escalation follows them as its own complete group.

### 5. Flashcards sanity check (1 min)

On a course where flashcards is **not** enabled, open `/local/ai_course_assistant/flashcards.php?courseid=<id>` — on Moodle 5.x this now shows a clean "Flashcards are not enabled for this course" page (pre-v6.1.1 it was a white-screen fatal). Enable flashcards for the course, generate a few from a Page/Book activity via the assistant, then reopen the review page to confirm the reveal + Again/Hard/Easy flow works.

### 6. Q&A (rolling)

- **"Does the emergency panel bypass the CLI?"** No — same class, same audit trail. The CLI remains the right tool when Moodle itself is unresponsive.
- **"Why is my Cached Tokens card missing?"** It appears once at least one post-v6.1.0 call reports cache data. Pre-v6.1.0 rows are NULL by design.
- **"Why did the flashcards page break before this patch?"** It called `print_error()`, removed in Moodle 5.0, on the not-enabled path. Fixed to throw `moodle_exception` like every other page.
- **"Can a non-admin reach the emergency panel?"** No — `moodle/site:config` capability plus sesskey on every action.
