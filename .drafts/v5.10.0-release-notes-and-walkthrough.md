# SOLA v5.10.0 — Release Notes, Feature Summary, and Admin Walkthrough

## Part 1: Release Notes (8 June 2026)

Makes SOLA correct and self-diagnosing on a small self-hosted backend (the single-GPU vLLM scenario, for example an NVIDIA A30 24GB), and tidies the project for handoff. The prompt builder, the metrics tuner, and a new self-test all share one window-to-chars ceiling, so a small `max_model_len` can no longer be overflowed, even in token-dense languages. Hosted large-context deployments are unaffected: leave the new window setting at 0 and nothing changes. No schema change.

- **Token-aware prompt budgeting.** A new **Backend context window** setting (`backend_context_tokens`, default 0 = hosted/unlimited) lets operators declare their backend `max_model_len`. When set, `context_builder::effective_budget_chars` shrinks the system-prompt character budget so the prompt plus reserved output and history fit the window, using conservative per-language chars-per-token divisors (`token_estimator`) so estimates round high and never overflow. The old hardcoded 60K Claude-specific cap is now a fallback only.
- **Bounded pre-first-byte retry.** A saturated backend that rejects with HTTP 429 or 503 is retried up to `backend_retry_attempts` times (default 2), honoring `Retry-After` capped at `backend_retry_max_wait` (default 5s) or a short exponential backoff. Retries happen only before any token streams; a mid-stream transport error is never retried, so visible output is never duplicated. A header function swallows the error body so a transient rejection forwards nothing to the learner.
- **Deployment presets.** A new `deployment_profile.php` admin page applies a recommended bundle for a self-hosted small-context backend (8192 window, 768 output, 6 history pairs, RAG top-k 4, retry 2) or restores hosted large-context defaults. Apply-once: the values are written into the normal settings and stay editable, so there is no hidden runtime override.
- **Backend self-test page.** A new `backend_selftest.php` runs a live check on demand: a tiny chat round-trip, auto-detection of the backend window via `/v1/models`, comparison against the configured value, the prompt-budget floor, and (when RAG is on) an embedding round-trip. Kept out of `health_check::run_all()` so post-upgrade structural checks stay network-free.
- **Metrics tuner respects the window.** The `prompt_budget_auto_tune` recommendation (which raises `prompt_budget_chars` to clear truncation) is now capped at the backend window ceiling, so it can no longer push the prompt past `max_model_len`.
- **Truncation-aware not-found answers.** When the course-content section is dropped or truncated and the learner is not on a specific page, the system prompt instructs the model to say it could not search the whole course and to suggest opening the specific page, rather than asserting the topic is absent. More relevant now that small backends clamp the budget.
- **Docs and handoff.** New [Deployment Sizing](Deployment-Sizing) wiki page (KV-cache-to-concurrency math, the A30 worked example, the lever list, the settings mapping), cross-linked from Admin Settings, RAG Guide, and Home. CLAUDE.md refreshed to current reality (version, RAG default, new key files, corrected source path).

### Backwards compatibility

Default-off: an install upgrades to v5.10.0 and behaves identically until an admin sets **Backend context window** above 0. Hosted deployments (Saylor production) are unaffected. The retry defaults (2 attempts) are conservative and only engage on transient 429/503.

### Testing

- New unit tests: `token_estimator_test`, `token_budget_clamp_test`, `prompt_budget_ceiling_test`, `provider_retry_test`, `truncation_hint_test`, `backend_probe_test`, `deployment_profile_test`. `health_check_test` confirms the structural checks remain network-free.
- Full plugin suite 398/398; validators 36/0; jailbreak 32/32; PHP lint clean on every modified file; lang-completeness green.
- i18n: the new English strings are admin-facing or LLM-instruction strings; 45-language translation runs through the lp-i18n pipeline (Moodle falls back to English for any not-yet-translated key in the interim).

## Part 3: Admin Walkthrough (self-hosted backend, ~15 min)

This walkthrough is the demo script for an admin standing up SOLA against a self-hosted backend (a single-GPU vLLM deployment, for example a 24GB card at an 8K window). Hosted-provider admins can skip it: leave **Backend context window** at 0 and SOLA behaves exactly as in v5.9.0.

### 0. Setup (1 min)

Have a self-hosted OpenAI-compatible backend reachable (vLLM, Ollama, or similar), its endpoint added to the SSRF allowlist (`ssrf_trusted_endpoints`), and the provider configured in Site administration → Plugins → Local plugins → AI Course Assistant. Note the backend's `max_model_len`: vLLM prints it at startup (`Maximum concurrency for N tokens per request`) and exposes it on `/v1/models`.

### 1. The problem this solves (2 min)

On a fixed GPU, the context window and the number of simultaneous learners trade directly against each other: `simultaneous requests = KV cache tokens / max_model_len per request`. An A30 with a 20,384-token KV cache at an 8K window serves about 2 learners at once. Before v5.10.0, SOLA assumed a 200K Claude window and budgeted the prompt in characters, so on a small backend the prompt could overflow `max_model_len` (worst in token-dense languages, where characters pack more tokens). v5.10.0 makes the prompt fit the window you actually have.

### 2. Declare your window (2 min)

Site administration → Plugins → Local plugins → AI Course Assistant → **AI Provider & Models** → **Backend context window (tokens)** (`backend_context_tokens`). Set it to your backend's `max_model_len`, for example 8192. Leave it at 0 for a hosted model. With it set, SOLA computes a token-aware ceiling on the system-prompt character budget (reserving room for the response and the conversation history) and uses conservative per-language token estimates so it rounds toward safety. The existing **System prompt character budget** still applies; the window clamp only ever makes it smaller.

### 3. Apply the preset instead (2 min)

Rather than tune knobs one by one, open the **Deployment presets** page (linked from settings, `deployment_profile.php`). It shows exactly what each preset writes. Click **Apply self-hosted small-context preset**: it sets the window to 8192, response length to 768, history to 6 pairs, RAG top-k to 4, chunk size to 300 words, and retry to 2 attempts, all written into the normal settings where they stay editable. The **Apply hosted large-context defaults** button restores the v5.9.0 baseline. There is no hidden override: the preset is a one-time write, so what you see in settings is what runs.

### 4. Self-test the backend (3 min)

Open the **Backend self-test** page (linked from settings, `backend_selftest.php`) and press **Run**. It performs a live check and reports pass / warn / fail for each row:
- **Chat round-trip** sends a tiny completion and reports latency.
- **Context window** attempts to read `max_model_len` from `/v1/models` and compares it to your configured value, warning on a mismatch or if clamping is off while the backend has a limit.
- **Prompt budget floor** confirms that, after reserving output, enough budget remains for the safety/identity sections (it fails if the window is so small the prompt cannot fit).
- **Embeddings** round-trips the embedding model when RAG is on, reporting the dimension.
Network calls run only when you press Run, so the page is safe to leave linked.

### 5. Why RAG is mandatory here (2 min)

A page of text is roughly 600 to 800 tokens, so a 250-page upload is about 160,000 tokens, far larger than the whole KV cache. It cannot go into the prompt at all. RAG (on by default) indexes the document once and retrieves only the handful of relevant chunks per question, so the prompt stays near 3K to 4K tokens regardless of source size. That keeps every request in the high-concurrency band and makes answers more specific. The preset sets RAG top-k to 4 so retrieval fits comfortably inside a small window.

### 6. Backpressure and the truncation hint (2 min)

When the backend is briefly saturated and rejects with 429 or 503, SOLA now retries up to **Backend retry attempts** times before any text streams, honoring a `Retry-After` header up to **Backend retry maximum wait**. The learner sees a momentary pause, not an error. A mid-stream failure is never retried, so output is never duplicated. Separately, when the course content has to be truncated to fit the window and the learner is not on a specific page, SOLA tells the model to answer "I could not search the whole course, try opening the specific page" rather than claiming the topic is absent.

### 7. Capacity planning, briefly (1 min)

Point to the new **Deployment Sizing** wiki page. The headline relation: halving the window roughly doubles the number of concurrent learners. The four hardware levers (lower `max_model_len`, fp8 KV cache, quantize weights, cap output) and the SOLA-side settings that mirror them are mapped in a table there, along with a fleet-sizing formula for scaling out across cards.

### 8. Q&A (rolling)

- **"Does this change anything for our hosted Saylor production?"** No. Backend context window defaults to 0, which is the old behavior exactly. This release is inert until an admin sets a window.
- **"What if the self-test cannot detect the window?"** Some backends do not expose `max_model_len` on `/v1/models`. The self-test warns and you set **Backend context window** manually from the value vLLM prints at startup.
- **"Will the retry slow down errors?"** Only transient 429/503 are retried, only before streaming, and the total added wait is capped by **Backend retry maximum wait**. A permanent error (auth, bad model) is surfaced immediately.
- **"Are the token estimates exact?"** No, they are conservative by design (they round high so the prompt never overflows). If you want extra margin, set the window slightly below the true value.

## Part 4: Admin-facing blurb

> **SOLA now sizes itself to your AI backend.** If you run a self-hosted model on a single GPU, tell SOLA your context window (`max_model_len`) in one setting, or apply the self-hosted preset, and SOLA keeps every prompt inside the window, even in token-dense languages. A new self-test page verifies your backend in one click, and SOLA now rides out a momentarily overloaded backend instead of erroring. Hosted deployments are unchanged. See the new Deployment Sizing guide for the context-window-versus-concurrency math.
