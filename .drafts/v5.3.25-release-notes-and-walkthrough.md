# SOLA v5.3.25

Pre-tag manual smoke checklist added and wired into the release process. Direct response to the bug classes that have shipped to production this year — buttons silently broken, JS-injected DOM misbehaving, AJAX silently failing, conditional UI sections drifting out of sync with feature flags. None of those classes can be caught by static checks.

## What's new

`.wiki/Release-Checklist.md` documents 9 critical-path items (~5 minutes total) plus an extended section with 6 area-specific smokes that run only when the matching code area changed.

Critical path:

1. **Drawer open + starters** — drawer opens, starters overlay renders, avatar SVG present, no console errors.
2. **Send a message + streaming response** — message streams token-by-token, four real SOLA_NEXT chips appear (no `chip1`, `||`, or empty buttons).
3. **Suggestion chip click** — chip click sends as user message, fresh response begins, new chip row appears.
4. **Reset + clear distinction** — home icon shows starters without clearing history; trash icon wipes history server-side.
5. **Quiz coach-mode block** — "just tell me the answer" gets refused, no literal correct option in response.
6. **GDPR self-service** — three buttons render and actually do something. Export downloads JSON; delete writes audit row.
7. **Help panel matches feature flags** — voice / analytics / attachments sections appear only when their header buttons are visible.
8. **Privacy notice link** — page loads, branding tokens resolve, no literal `Saylor` placeholder if rebranded.
9. **Console clean** — no `Uncaught`, no 4xx/5xx XHRs except deliberate aborts, no Mustache warnings.

Extended (only when the area changed): voice mode, course settings save, analytics dashboard, mobile viewport, per-language i18n, RAG admin.

The checklist documents its own skip rules: CSS-only releases skip 5–8, i18n-only releases skip 5–6 and run only the language item, test-only releases skip the whole list.

## Why now

Of the 12 production bugs caught manually on Catalyst staging or by user reports during v5.3 development, the test suite caught zero. The suite caught the structural ones (privacy provider gaps, missing lang strings, contract mismatches) but every UI-behaviour bug — buttons that ran no JS, chip placeholders, voice mode missing course context, help panel referencing disabled features, GDPR buttons silently no-op — only surfaced when a human clicked the thing. A 5-minute click-through before tag is the cheapest possible coverage for that class.

## Wired in

`CLAUDE.md` release-checklist section now lists the smoke run as step 5 (after validators, before tag). The validator suite exit message refers to the smoke as the next step. No CI gate added — this is intentionally human-in-the-loop, since the bugs it catches are the ones automation can't.

## Backwards compatibility

Drop-in upgrade from v5.3.24. No code changes, no test changes, no schema changes.
